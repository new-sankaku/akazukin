## 2026-03-14: N+1クエリ問題の修正パターン
- `TeamUseCase.listTeams()` でチームごとにメンバーを1件ずつ取得していた (N+1)
- `toResponseDto()` でメンバーごとにユーザーを1件ずつ取得していた (N+1)
- 修正: ポートインターフェースにバッチ取得メソッド (`findMembersByTeamIds`, `findAllByIds`) を追加し、IN句で一括取得するようにした
- 既存の単一取得の `toResponseDto(Team, List<TeamMember>)` は内部で `fetchUsersById` を呼ぶオーバーロードとして残し、`createTeam`, `addMember`, `getTeam` など単一チーム操作の呼び出し元は変更不要にした
- テスト用のインメモリ実装 (`InMemoryUserRepository`) にも新メソッドの実装を忘れずに追加する必要がある

## 2026-03-14: SDK層とAdapter層の二重リトライ問題
- `AbstractSnsAdapter.checkResponseStatus()` が 429 レスポンスに対して `Thread.sleep()` で待機してから `RetryableRateLimitException` を投げていた
- SDK層（例: `TwitterClient`）も独自にリトライ/バックオフを行うため、両層で待機が発生し待ち時間が倍以上になる問題があった
- 修正: Adapter層の `checkResponseStatus()` から `Thread.sleep()` を削除し、`RateLimitExceededException` を即座に投げるように変更
- リトライ/バックオフの責務はSDK層に統一する方針
- 注意: 現状 BlueskyAdapter, MastodonAdapter, RedditAdapter, PinterestAdapter, TelegramAdapter, InstagramAdapter, ThreadsAdapter, VkAdapter は SDK クライアントを使わずAdapter層で直接 HttpClient を使用している。これらはSDK層のリトライの恩恵を受けないため、将来的にSDKクライアントへの移行を検討する
- `PostPublishUseCase.isRetryable()` の `RetryableRateLimitException` クラス名チェックは、adapter層の catch ブロックで `SnsApiException` に包まれるため実質的にデッドコードだった

## 2026-03-14: TikTok API の制約と実装上の注意点
- **テキストのみ投稿は不可** — TikTokは動画/写真が必須。Content Posting API にテキストオンリーのエンドポイントは存在しない
- **投稿削除APIなし** — TikTok APIは投稿の削除をサポートしていない。`deletePost()` は `UnsupportedOperationException` をスローする設計にした
- **未審査アプリは非公開投稿のみ** — TikTok Developer Portalでアプリ審査を通過するまで、全投稿が `SELF_ONLY`（自分だけに表示）に制限される。デフォルトの privacy_level を `SELF_ONLY` に設定済み
- **アクセストークンの有効期限が短い (24時間)** — refreshToken (365日有効) での定期更新が必須。他のSNS (Twitter等) より有効期限が短いため、トークンリフレッシュの頻度を考慮する必要がある
- **認証フローはOAuth2** — ただしPKCEは不要 (Twitterとは異なる)。client_key + client_secret + authorization code でトークン交換
- **TikTok APIの認証情報の呼び名が独特** — 他のSNSが `client_id` / `client_secret` を使うのに対し、TikTokは `client_key` / `client_secret` を使う。環境変数名は `TIKTOK_CLIENT_KEY` / `TIKTOK_CLIENT_SECRET` にした
- **レスポンス構造が他のSNSと異なる** — TikTokのAPIレスポンスは `{"data": {"user": {...}}}` のようにネストが深い。エラーレスポンスも `{"error": {"code": "...", "message": "...", "log_id": "..."}}` という独自構造
- **投稿はpublish_idで管理** — 投稿後にURLは即座に取得できず、`publish_id` が返される。投稿ステータスの確認は別途 `/post/publish/status/fetch/` で行う必要がある
- **公式Java SDKは存在しない** — tiktok-business-api-sdk は広告API専用。Content Posting APIには独自SDK (akazukin-sdk-tiktok) を実装した

## 2026-03-14: 保守機能一括実装で得た知見

### Worktree並行開発のマージ戦略
- 6チームをgit worktreeで並行実装すると、`build.gradle.kts` と `application.properties` で大量の競合が発生する
- 競合が少ないブランチ（新規ファイル中心）から先にマージし、設定ファイルを触るブランチは後回しにするのが効率的
- worktreeディレクトリは `.gitignore` に追加しておかないと、git addで埋め込みリポジトリとしてコミットされる事故が起きる

### Micrometerメトリクスのタグ設計
- `cause.getClass().getSimpleName()` を直接タグに使うとカーディナリティ爆発（未知の例外クラスが無限にタグを増やし、PrometheusでOOM）
- 対策: 既知例外のホワイトリストを定義し、未知例外は "other" に正規化する
- success/failure で同じメトリクス名を使う場合、タグセットを揃える必要がある（success側にも `tag("error", "none")` を追加）
- Counter/Timerインスタンスは毎回 `builder().register()` せず、フィールドにキャッシュして再利用する

### SmallRye Fault Toleranceの落とし穴
- `@CircuitBreaker` / `@Retry` はCDI interceptorとして動作するため、CDI管理外のインスタンス（`new` で生成）ではアノテーションが完全に無視される
- SmallRye Fault Toleranceのサーキットブレーカー名は `FQクラス名/メソッド名` で管理される。`CircuitBreakerMaintenance.currentState("カスタム名")` で取得しても、アノテーションで作られたものとは別物になる
- 自作のサーキットブレーカー（InMemoryCircuitBreakerRegistry）とアノテーションベースのサーキットブレーカーを共存させると、二重チェック・二重カウントが発生する。どちらか一方に統一すべき

### レート制限（Bucket4j）のメモリ管理
- ConcurrentHashMapでユーザー別バケットを管理すると、ユーザー数増加でOOM
- 対策: Caffeineキャッシュに置き換えて `maximumSize` + `expireAfterAccess` を設定するか、定期クリーンアップを実装

### バックグラウンドジョブ（@Scheduled）の排他制御
- `@Scheduled` は各Quarkusインスタンスで独立実行される。複数インスタンス環境では同一ジョブが並行実行され、二重投稿等の問題が発生する
- 対策: `concurrentExecution = SKIP`（Quarkus組み込み）。本格的にはDBレベルの `SELECT ... FOR UPDATE SKIP LOCKED` や分散ロックが必要

### Redis KEYSコマンドは本番禁止
- `KEYS pattern` はO(N)でRedis全体をブロッキングスキャンする
- 本番環境では `SCAN` コマンドに置き換えるか、パターン指定の一括操作を避ける

### 通知のCompositeパターンでの注意
- CompositeNotificationSender で `@Alternative @Priority(1)` を使うと、元のDbNotificationSenderが完全に置き換わる
- DB保存（必須）+ Email/Webhook（補助）という優先度の異なる処理を混ぜる場合、DB保存は失敗時にリスロー、補助は失敗をログに記録して続行、という区別が必要

### Webhook送信のSSRF対策
- 外部URLにHTTPリクエストを送る機能には必ずSSRF対策が必要
- プライベートIP（10.x, 172.16-31.x, 192.168.x）、リンクローカル（169.254.x）、ループバック（127.x, ::1）をブロックする
- AWS環境ではメタデータエンドポイント（169.254.169.254）経由でIAMクレデンシャルが窃取される

### application.propertiesのセキュリティ
- デフォルトプロファイルにDB認証情報や暗号化キーのデフォルト値を置くと、環境変数未設定時にそのまま本番で使われる
- 対策: `%prod.quarkus.datasource.password=${DB_PASSWORD}` のようにデフォルト値なしにして、未設定なら起動失敗させる

### ヘルスチェックでの情報漏洩
- DatabaseHealthCheckで `connection.getMetaData().getURL()` を返すと、JDBC URLにホスト名やポートが含まれ外部に漏洩する
- 本番では接続成功/失敗とDB製品名のみ返す
